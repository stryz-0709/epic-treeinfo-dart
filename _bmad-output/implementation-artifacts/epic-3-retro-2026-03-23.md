## Highlight 1: Role-Scope and Security Correctness

| Story | Security/Scope Evidence | Assessment |
|---|---|---|
| `3-1` | `GET /api/mobile/incidents` enforces ranger self-only and leader authorized scope server-side; read-only contract preserved (no create/edit); defensive guards for malformed rows/timestamps and cursor validation. | Strong server-side authorization and read-surface hardening. |

### Security Correctness Themes

1. **Server-side enforcement remained the source of truth** across incidents and schedules.
2. **Read-only incident boundary held cleanly** with no exposed create/edit controls in Phase 1 scope.
3. **Leader-only schedule mutation stayed enforced** through explicit role guards and negative-path regression tests.
4. **Defensive normalization and malformed-input handling** reduced scope bypass risk from dirty payloads and edge inputs.

## Highlight 2: Schedule Workflow Reliability

### Reliability Evidence from Stories `3-3` and `3-4`

- API reliability hardening:
  - Date/filter validation and deterministic schedule response metadata
  - Concurrency-safe writes with dedicated locking
  - Snapshot-safe reads and audit timestamp fallback behavior for malformed legacy values
  - Note normalization and payload guardrails to prevent unstable write behavior
- UX reliability hardening:
  - In-flight submit guard to prevent duplicate leader writes
  - Month-navigation/load race handling to avoid stale state overwrite
  - Stable selected-value handling for leader ranger filters
  - Clear loading/empty/error pathways with localized messaging

### Reliability Assessment

- Schedule workflows are **operationally reliable** under normal and adversarial interaction paths.
- Remaining reliability risk is **integration-surface drift** when Epic 4 cache/incremental refresh behavior expands; this is manageable with explicit contract tests and replay-safe invariants.

## What Went Well

- Adversarial review loops were executed to closure and converted into concrete fixes.
- Regression discipline (targeted + full suites) remained active on both backend and mobile.
- Story artifacts recorded meaningful hardening notes and traceable test evidence.
- Security and scope boundaries were treated as acceptance gates, not optional polish.

## Challenges and Friction

- Some reliability issues emerged only after deeper review cycles (cursor contract nuances, concurrent write behavior, async UI races).
- Cross-layer consistency (backend metadata ↔ provider rendering semantics) required repeated normalization work.
- Defensive behavior had to be standardized repeatedly across stories instead of being inherited from a shared contract library.

## Key Lessons

1. **Role-scope correctness should be codified as reusable contract tests** and applied to every protected endpoint by default.
2. **Schedule reliability depends on both API invariants and UI state-machine stability**; either side alone is insufficient.
3. **Adversarial loops are most effective when triage outcomes are explicitly captured and retested end-to-end.**
4. **Operational-state UX (loading/stale/error/empty) directly contributes to reliability, not just user experience quality.**

## Technical Debt and Carry-Forward Items

- Shared contract helpers for scope metadata normalization are still fragmented.
- Defect-prevention evidence is present but not yet in a single standardized section format across all story artifacts.
- Cross-surface schema drift checks (backend response shape vs mobile models) should be formalized as a standing gate.
