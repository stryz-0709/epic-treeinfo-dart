# Epic 1 Retrospective — Secure Mobile Access & Role-Safe Authentication

- **Epic:** 1
- **Stories reviewed:** 1-1, 1-2, 1-3, 1-4
- **Retrospective date:** 2026-03-23
- **Facilitation style:** concise, no-blame, systems-focused

## Epic 1 outcome snapshot

Epic 1 established a production-grade authentication and authorization baseline for mobile access:
- Backend login + refresh/logout flows implemented and validated.
- Role-safe server-side guards enforced across protected mobile endpoints.
- Security baseline hardened (secret hygiene checks + production config safeguards).
- Regression coverage expanded and repeatedly validated during review hardening.

## Top lessons (no-blame)

1. **Early security hardening loops pay off.**
   Adversarial review surfaced medium/low issues that were inexpensive to fix once caught early (input bounds, timing-safe checks, malformed session handling).

2. **Fail-closed parsing prevents broad outages.**
   Small malformed data defects (e.g., invalid persisted `day_key`) can break entire responses unless parsing is defensive and skips bad records safely.

3. **Auth reliability depends on lifecycle edge cases, not just happy paths.**
   Token invalidation, expired/invalid refresh handling, and whitespace/malformed payload rejection were key to making session behavior predictable.

4. **Secret hygiene must be automated, not manual.**
   Runtime guards + repository scanning + startup validation were necessary to prevent privileged secret drift into mobile surfaces.

## Concrete action items

1. **Create a reusable mobile-auth edge-case test matrix** (invalid, expired, malformed, whitespace tokens).
   - **Owner:** Dev + QA
   - **Success criteria:** Matrix exists in test docs and is covered by automated tests run in CI.

2. **Standardize defensive row parsing for mobile summary endpoints.**
   - **Owner:** Dev
   - **Success criteria:** Shared parsing helper/pattern adopted; malformed row regressions have dedicated tests.

3. **Keep security baseline checks as a required pre-merge gate.**
   - **Owner:** Platform/Security
   - **Success criteria:** Secret scanner + production config validation are required in the default verification flow.

4. **Publish role-scope API contract examples for leader vs ranger.**
   - **Owner:** PO + Dev
   - **Success criteria:** Contract examples are documented and referenced by both backend and mobile implementation stories.

## Next-epic prep risks (Epic 2)

1. **Role-scope drift risk in Work Management views.**
   - Epic 2 introduces leader ranger-filter and ranger self-only calendar views; any scope regression can leak data.

2. **Day-boundary/idempotency risk for check-ins.**
   - Epic 2 depends on strict `user_id + day_key` behavior and project timezone consistency; mismatches can create false duplicates or missed check-ins.

3. **Performance risk on calendar summary queries.**
   - Leader/team-scoped calendar requests may grow quickly; pagination/filtering behavior must be enforced from the start.

4. **Security-regression risk during mobile UX iteration.**
   - Rapid mobile changes in Epic 2 can accidentally reintroduce forbidden env/config patterns unless scanning remains mandatory.

## Epic 2 readiness recommendations

- Treat role-scope tests as release-blocking for Work Management endpoints.
- Lock timezone/day-key policy tests before implementing app-open check-in trigger.
- Verify pagination/filtering behavior with representative leader-scope datasets.
- Keep secret-hygiene checks in every backend/mobile validation run.

## Close

Epic 1 delivered a strong security and auth foundation. The biggest win was turning potential failure modes into explicit guardrails and tests. If Epic 2 starts with those guardrails intact, delivery risk should stay controlled even as scope broadens into calendar + check-in behavior.
